# NoTIP Cryptographic Contract

**Status:** Authoritative
**Version:** 1.0
**Scope:** End-to-end encryption of BLE telemetry from IoT gateway to client

---

## Overview — Opaque Pipeline (Regola Zero)

**No cloud component ever decrypts telemetry payloads.**

Data flows as encrypted blobs from the gateway through NATS → Data Consumer → TimescaleDB → Data API, and is decrypted only by:
- The **browser** (Angular Web Worker via WebCrypto API), or
- An **authorized API client** that independently decrypts after fetching the key from the Management API.

```
Gateway          NATS          Data Consumer    TimescaleDB    Data API     Browser / API Client
  │               │                  │                │             │               │
  │──[encrypt]──► │──[ciphertext]──► │──[ciphertext]──►│             │               │
  │               │                  │                │ ◄──query──  │               │
  │               │                  │                │ ──ciphertext─►│               │
  │               │                  │                │             │──[ciphertext]──►│
  │               │                  │                │             │               │──[decrypt]─►plaintext
```

This is "server-assisted E2EE": the platform stores AES keys in the Management DB (encrypted at rest), but never uses them operationally to read data in transit. The security boundary is the Management DB + `DB_ENCRYPTION_KEY`.

---

## 1. Key Hierarchy

### 1.1 Gateway AES Key

Each provisioned gateway has a unique **256-bit AES key** generated by the Provisioning Service at provisioning time (`crypto.randomBytes(32)` or equivalent CSPRNG).

- **Generated by:** Provisioning Service during the `/provision/onboard` flow
- **Stored by:** Management API in the `encryption_keys` table, encrypted at rest with `DB_ENCRYPTION_KEY` (AES-256-GCM)
- **Delivered to gateway:** once, over HTTPS at provisioning time — never re-transmitted
- **Scope:** one key per gateway; all sensors on a gateway share the same key
- **Revocation:** set `revoked_at` on decommissioning; key delivery endpoint returns 403 for revoked keys

### 1.2 DB Encryption Key

The `DB_ENCRYPTION_KEY` wraps all gateway AES keys stored at rest.

- 256-bit key (32 bytes), hex-encoded
- Stored as Docker secret `db_encryption_key` → `/run/secrets/db_encryption_key`
- Used by the Management API only: decrypt key material immediately before returning it in an HTTP response; the plaintext key exists in memory only for the duration of that response
- **Loss of this key renders all stored gateway keys unrecoverable.** Back it up separately from the database.

### 1.3 Key Versioning

Every key stored has an integer `key_version`. This value is included in every telemetry envelope so that clients can detect a key rotation (post-MVP feature) without silent decryption failure. At MVP, `key_version` is always `1`; the field exists for forward compatibility.

## 2. Telemetry Envelope Format

Every telemetry message published on NATS, stored in TimescaleDB, and returned by the Data API has this exact JSON structure:

```json
{
  "gatewayId":     "<UUID v4>",
  "sensorId":      "<UUID — self-declared by firmware, not registered in DB>",
  "sensorType":    "temperature|humidity|movement|pressure|biometric",
  "timestamp":     "<ISO 8601 UTC>",
  "keyVersion":    1,

  "encryptedData": "<base64 — AES-256-GCM ciphertext of the sensor reading>",
  "iv":            "<base64 — 12 bytes random nonce>",
  "authTag":       "<base64 — 16 bytes GCM authentication tag>"
}
```

### Plaintext fields (indexable metadata)

| Field | Type | Description |
|-------|------|-------------|
| `gatewayId` | UUID v4 string | Gateway that produced the data |
| `sensorId` | UUID v4 string | BLE peripheral/sensor identifier, self-assigned by firmware. Not validated against any DB table. A sensor can move between gateways |
| `sensorType` | enum string | `temperature`, `humidity`, `movement`, `pressure`, `biometric` |
| `timestamp` | ISO 8601 UTC | Gateway-local measurement time |
| `keyVersion` | integer | AES key version. Clients use this to detect key rotation |

### Encrypted fields (opaque to all cloud components)

| Field | Encoding | Description |
|-------|----------|-------------|
| `encryptedData` | Base64 | AES-256-GCM ciphertext of the raw sensor reading value |
| `iv` | Base64 | 12-byte (96-bit) random nonce, generated fresh per message |
| `authTag` | Base64 | 16-byte (128-bit) GCM authentication tag |

The plaintext content of `encryptedData` (after decryption) is the raw sensor reading: a byte sequence whose interpretation is sensor-type-specific and defined by the gateway firmware.

---

## 3. Encryption Parameters

| Parameter | Value |
|-----------|-------|
| Algorithm | AES-256-GCM |
| Key length | 256 bits (32 bytes) |
| IV (nonce) | 96 bits (12 bytes), randomly generated **per message** |
| Auth tag | 128 bits (16 bytes), output of AES-GCM |
| AAD | None (not used in this implementation) |

---

## 4. NATS Transport

Messages are published to `telemetry.data.{tenantId}.{gatewayId}`.

The entire JSON object from §2 is the NATS message payload. The plaintext fields (`gatewayId`, `sensorId`, `sensorType`, `timestamp`, `keyVersion`) are needed for routing, stream retention, and Data API filtering. They are not encrypted.

---

## 5. Key Delivery (Management API → Client)

Clients fetch AES keys from the Management API over HTTPS (via Nginx):

- `GET /api/mgmt/keys?gateway_id={gatewayId}` — single gateway key
- `GET /api/mgmt/keys` — all active keys for the caller's tenant

Rules enforced by the Management API:
1. `tenant_id` in JWT must match `tenant_id` of the gateway in the DB. Cross-tenant → 403.
2. If `revoked_at` is set on the key (gateway decommissioned) → 403.
3. If the JWT contains an `act` claim (admin impersonation session) → 403. An admin impersonating a user must not be able to exfiltrate keys on their behalf.
4. The `key_material` is decrypted from the DB using `DB_ENCRYPTION_KEY` immediately before the response is written. The plaintext key is never logged.

Response format:
```json
[
  {
    "gatewayId":   "<UUID>",
    "keyMaterial": "<base64 AES-256 key — 32 bytes>",
    "keyVersion":  1
  }
]
```

---

## 6. Client Decryption (Browser / API Client)

1. Fetch the gateway's AES key from `GET /keys?gateway_id=...`
2. In the browser: pass key bytes to the Web Worker via `postMessage()`, import with `crypto.subtle.importKey(..., { extractable: false })`. The raw bytes are discarded after import.
3. For each envelope received from the Data API:
   - Base64-decode `iv`, `encryptedData`, `authTag`
   - AES-256-GCM decrypt with the CryptoKey, IV, and auth tag
   - Verify authentication tag — reject any message where tag verification fails
   - Check `keyVersion` matches the cached key version. If mismatch: emit `{ type: "key_version_mismatch", cachedVersion: N, payloadVersion: M }` to the main thread (browser reloads and re-fetches key)

---

## 7. Security Considerations

- **Nonce uniqueness:** The gateway MUST generate a fresh 12-byte random nonce per message. Nonce reuse under the same key breaks GCM security entirely (key recovery becomes feasible).
- **Tag verification:** Implementations MUST verify the GCM auth tag before using plaintext. Silent failures are not acceptable.
- **Key material in logs:** The gateway AES key and `DB_ENCRYPTION_KEY` must never appear in application logs, error messages, or stack traces.
- **DB_ENCRYPTION_KEY rotation:** Not supported in the MVP. Rotation requires re-encrypting all stored gateway keys. Plan before scaling to production.
- **Gateway compromise:** Re-provision the gateway to rotate its AES key. The old key remains in the DB for historical data decryption until explicitly removed.
- **Impersonation guard:** The `act` claim check on key delivery endpoints is mandatory. See §5 rule 3.
